# Matrice d'attaque

Ce document recense tous les vecteurs d'attaque connus contre l'infrastructure NixOS sur MacBook Pro M1 Pro, classés par contexte opérationnel. Chaque vecteur est accompagné de sa description, de ses mitigations et de son statut.

**Comment lire cette matrice :**

- **Actif** : le vecteur est réaliste et mitigé (mais pas éliminé)
- **Éliminé** : le vecteur est rendu physiquement ou techniquement impossible
- **Accepté** : le risque résiduel est connu, documenté et formellement accepté (voir [RISQUES-RESIDUELS.md](RISQUES-RESIDUELS.md))

Les mitigations renvoient aux ADR pour les justifications détaillées.

-----

## Catégorie 1 — Machine déverrouillée et en fonctionnement

| Vecteur                        | Description                                                                                                                                                                                   | Mitigation                                                                                                                                            | Statut  |
| ------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- | ------- |
| Exploit kernel direct          | Vulnérabilité kernel (buffer overflow, use-after-free dans un driver) → exécution ring 0 → lecture de la clé LUKS depuis les structures dm-crypt en mémoire kernel. Vecteur le plus réaliste. | Kernel à jour, surface d'attaque réduite (pas de services inutiles), AppArmor sur les services exposés, `lockdown=confidentiality` (Phase 1).        | Actif   |
| Attaque DMA via Thunderbolt    | Appareil malveillant branché sur USB-C/Thunderbolt tente de lire la RAM via DMA.                                                                                                             | DART (Device Address Resolution Table, IOMMU Apple Silicon) filtre les accès DMA par périphérique via device tree. Chaque périphérique dans sa propre « cage » mémoire. `CONFIG_APPLE_DART=y`, actif par défaut. Ne pas brancher d'appareils inconnus. | Actif   |
| Shutdown nocturne malveillant  | Attaquant root modifie `configuration.nix` pour retirer le lockdown, fait un `nixos-rebuild`, déclenche un shutdown à 3h. L'utilisateur saisit la passphrase LUKS au réveil croyant à un crash. | Vérification des hashes de la partition EFI depuis macOS après tout reboot inattendu, monitoring de l'uptime, `safe-rebuild.sh` (voir [ADR 0007](adr/0007-nixos-rebuild-ne-verifie-pas-signatures.md)). | Actif   |

## Catégorie 2 — Machine en veille (s2idle)

| Vecteur                          | Description                                                                                                                                                                                                          | Mitigation                                                                                                                                                                                          | Statut   |
| -------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------- |
| Clé LUKS en RAM pendant la veille | s2idle maintient la RAM alimentée, la clé LUKS est présente. Le risque en s2idle est un sous-ensemble strict du risque en fonctionnement normal — même clé en RAM, mais espace utilisateur gelé (surface réduite). | `lockdown=confidentiality` bloque l'extraction userspace, DART bloque le DMA, SiP élimine le cold boot. Shutdown automatique après 30 min borne l'exposition. Voir [ADR 0002](adr/0002-hibernation-ecartee-s2idle-shutdown.md). | Accepté  |

## Catégorie 3 — Machine éteinte (shutdown complet)

| Vecteur                                | Description                                                                                                                                                                                                                                    | Mitigation                                                                                                                                                       | Statut  |
| -------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------- |
| Remplacement kernel/initrd sur EFI     | Attaquant avec accès physique modifie le kernel ou l'initrd sur la partition EFI non chiffrée. Au prochain boot, le kernel/initrd piégé capture la clé LUKS ou les credentials FIDO2.                                                          | Vérification des hashes depuis macOS (SSV garantit l'intégrité de l'environnement de vérification), signature du kernel avec clé SEP. Voir [ADR 0011](adr/0011-boot-non-chiffrable-fido2.md).     | Actif   |
| Cold boot                              | Tentative de refroidir la RAM pour en extraire le contenu. RAM LPDDR5 intégrée au SiP — dessoudage = chaleur = destruction des données. Le froid nécessaire à la préservation et la chaleur nécessaire à l'extraction sont mutuellement exclusifs. Aucune recherche publiée, communauté forensique tournée vers l'acquisition logique. | Éliminé par la physique (architecture SiP Apple Silicon). Voir [ADR 0010](adr/0010-modele-menace-et-securite-apple-silicon.md).                                  | Éliminé |
| Dessoudage des puces NAND              | Extraction physique des puces de stockage. Sans la clé LUKS (disparue de la RAM par shutdown), l'attaquant obtient du bruit : chiffrement hardware permanent Apple (AES-256, lié au SoC) + LUKS2 par-dessus. Double chiffrement.               | Bruteforce impossible sur double couche AES-256. Voir [ADR 0010](adr/0010-modele-menace-et-securite-apple-silicon.md).                                           | Éliminé |
| DFU + réinstallation (evil maid avancée) | Mode DFU pour effacer et réinstaller un OS piégé.                                                                                                                                                                                             | Find My Mac / Activation Lock bloque le Setup Assistant. Données précédentes détruites cryptographiquement (pas de récupération pour l'attaquant). Mac vierge = signal d'alerte évident.            | Actif   |

## Catégorie 4 — Chaîne de déploiement

| Vecteur                              | Description                                                                                                                                             | Mitigation                                                                                                                                                                                                    | Statut |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------ |
| Push malveillant sur GitHub          | Attaquant obtient un accès push au repo, modifie `configuration.nix`. Le prochain `nixos-rebuild` déploie la config malveillante.                       | Clé SSH sk résidente pour auth GitHub (push impossible sans YubiKey + touch). `safe-rebuild.sh` vérifie la signature du commit. Voir [ADR 0007](adr/0007-nixos-rebuild-ne-verifie-pas-signatures.md).         | Actif  |
| Compromission de GitHub              | Employé malveillant, faille infra, injonction judiciaire — modification silencieuse du repo.                                                            | `safe-rebuild.sh` vérifie la signature (commit injecté non signé par la clé sk). Limite : replay d'un ancien commit signé non détecté. `--ff-only` et log d'audit atténuent.                                  | Actif  |
| Rebuild local malveillant            | Attaquant root pointe `nixos-rebuild` vers un flake arbitraire.                                                                                         | `safe-rebuild.sh` comme convention + log d'audit. Root peut contourner → vérification des hashes depuis macOS post-reboot + signature kernel via SEP/Touch ID.                                                | Actif  |
| Empoisonnement du flake.lock         | Modification de `flake.lock` pour pointer vers un nixpkgs malveillant, commit signé avec une clé compromise.                                           | Revue humaine du diff de `flake.lock` avant chaque commit de mise à jour nixpkgs. Le wrapper ne détecte pas ce vecteur automatiquement — limite documentée.                                                   | Actif  |
| nixos-rebuild sans vérification      | Comportement natif : `git fetch` + build aveugle, aucune vérification de signature.                                                                    | `safe-rebuild.sh` comble ce trou. Voir [ADR 0007](adr/0007-nixos-rebuild-ne-verifie-pas-signatures.md).                                                                                                      | Actif  |

## Catégorie 5 — Supply chain et hors machine

| Vecteur                                   | Description                                                                                                                      | Mitigation                                                                                                                                                        | Statut |
| ----------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------ |
| Compromission kernel/initrd via build chain | Paquet NixOS ou dépendance contient un rootkit kernel.                                                                           | `flake.lock` épinglé, commits signés, reproductibilité des builds NixOS.                                                                                         | Actif  |
| Compromission du serveur de sauvegarde    | Sauvegardes non chiffrées côté client → l'attaquant contourne toute la sécurité du Mac.                                         | Chiffrement restic/borg côté client avec clé sur YubiKey.                                                                                                        | Actif  |
| Ingénierie sociale / coercition           | Convaincre ou contraindre le propriétaire de déverrouiller. Aucune mitigation technique — problème humain.                       | Multi-facteur (objet + biométrie/PIN + passphrase) rend la coercition plus complexe. Voir [ADR 0004](adr/0004-luks-imbrique-passphrase-externe-yubikey-interne.md). | Actif  |
